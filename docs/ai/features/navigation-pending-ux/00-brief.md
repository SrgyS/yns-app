# Brief: Navigation Pending UX (Next.js App Router)

## Название задачи
Единый pending UX для навигации в приложении на Next.js App Router

## Контекст
Сейчас при переходах между страницами пользователь не всегда получает понятную обратную связь. В момент навигации интерфейс может выглядеть как "ничего не произошло", особенно на медленных страницах или при server data loading. Это повышает вероятность повторных кликов, ухудшает восприятие скорости приложения и делает UX менее современным.

## Цель
Сделать единый, аккуратный и современный pending UX для внутренних переходов и action-based navigation без fullscreen loader по умолчанию.

## Что нужно решить
- показать пользователю, что навигация началась
- убрать ощущение зависания интерфейса
- снизить количество повторных кликов
- добавить skeleton для data-heavy маршрутов
- реализовать решение в стиле App Router, без deprecated router events

## Что будет реализовано

### 1. Глобальный top progress bar
Один общий индикатор прогресса, который монтируется в `app/layout.tsx`.

Функции:
- стартует не сразу, а с задержкой, чтобы не было мигания на быстрых переходах
- показывает псевдо-прогресс до 90% в процессе ожидания
- завершается и скрывается после завершения навигации
- не влияет на layout и не вызывает сдвиг страницы

### 2. SmartLink
Обертка над `next/link`, которая автоматически запускает top progress bar при внутренней навигации.

Что дает:
- единое поведение ссылок по всему приложению
- отсутствие дублирования логики
- сохранение стандартного поведения Next.js Link

### 3. Navigation feedback helpers
Отдельные утилиты для запуска и завершения navigation feedback без прямого обращения к `window` из feature-кода.

Функции:
- `startNavigationFeedback()`
- `doneNavigationFeedback()`

Что дает:
- более чистую архитектуру
- меньше связности
- удобство повторного использования

### 4. Паттерн для кнопок навигации
Для кнопок, которые переводят пользователя на другой route:
- использовать `useTransition`
- вызывать `router.push()`
- сразу блокировать кнопку
- показывать inline spinner
- запускать progress bar при клике

Что дает:
- мгновенный отклик в месте действия
- защиту от двойного клика
- предсказуемый UX

### 5. Route skeletons через `loading.tsx`
Для тяжелых маршрутов будут добавлены route-level skeleton экраны.

Требования:
- skeleton должен быть похож на реальную структуру страницы
- не должно быть пустого белого экрана при ожидании данных
- использовать Tailwind utility-классы и поддерживать обе темы

## Уточнение для mobile bottom navigation

Если в приложении используется нижнее мобильное меню с иконками, его pending UX должен отличаться от обычных текстовых ссылок.

### Требования для mobile bottom nav
- При тапе по иконке пользователь должен сразу получать локальный визуальный отклик: active press state, изменение opacity, background или scale.
- Если нажат пункт, ведущий на другой route, должен запускаться global top progress bar по той же схеме с delay.
- Если нажат уже активный пункт меню, повторная навигация не должна запускаться.
- Для already active tab progress bar не должен стартовать.
- Внутри самой иконки не нужно показывать spinner в стандартном сценарии.
- Нижняя навигация не должна полностью блокироваться на время обычного route transition.
- Для тяжелых экранов после перехода должен использоваться route-level skeleton через `loading.tsx`.

## UX-правило
Для mobile bottom nav использовать паттерн:
- tap feedback
- delayed top progress bar
- skeleton на целевом route при медленной загрузке

### Что важно не делать
- не показывать fullscreen loader при обычном переключении между табами
- не заменять иконку loader-спиннером без особой необходимости
- не запускать navigation feedback при клике по уже открытому разделу
- не вызывать лишний `router.push()` на текущий pathname

### Рекомендация по реализации
Для нижнего меню лучше использовать отдельный компонент, например `MobileBottomNavItem`, а не применять обычный `SmartLink` без дополнительной логики.

Внутри компонента должно быть:
- сравнение target route с текущим pathname
- если route уже активен -> не запускать progress и не выполнять navigation
- если route новый -> запускать `startNavigationFeedback()` и выполнять переход
- active state и pending feedback должны быть разделены логически

### Acceptance criteria для mobile bottom nav
- тап по неактивной иконке дает мгновенный визуальный отклик
- при реальной задержке перехода появляется top progress bar
- тап по активной иконке не запускает progress и повторную навигацию
- иконки не теряют читаемость из-за inline spinner
- нижнее меню остается стабильным и не дергается во время переходов

## UX-логика по времени ожидания
- 0-100ms -> ничего не показывать
- 100-500ms -> мягкий feedback, например top progress bar или pending state кнопки
- 500-1500ms -> skeleton через `loading.tsx`
- более 1500ms -> при необходимости можно добавить текстовую подсказку внутри зоны skeleton

## Что не входит в задачу
- fullscreen loader для всех переходов
- глобальные спиннеры без контекста
- использование deprecated router events из Pages Router

## Ожидаемый результат после выполнения
Пользователь будет видеть понятную и современную обратную связь при навигации:
- при клике по ссылке появится аккуратный global progress indicator
- при клике по action button пользователь сразу увидит disabled state и loader
- на тяжелых страницах вместо пустоты будет skeleton
- интерфейс будет ощущаться быстрее и стабильнее

## Польза для продукта
- выше perceived performance
- ниже вероятность повторных кликов
- меньше ощущения "ничего не произошло"
- более качественный и современный UX
- единый стандарт поведения навигации во всем приложении

## Польза для разработки
- reusable shared-инфраструктура
- совместимость с Next.js 14+ App Router
- корректный подход под React 18, Suspense и streaming
- чистая архитектура без привязки к конкретной бизнес-сущности
- удобная база для масштабирования на новые страницы и сценарии

## Ключевые deliverables
- `app/layout.tsx` -> подключение `<TopProgressBar />`
- `shared/ui/top-progress-bar.tsx`
- `shared/ui/smart-link.tsx`
- `shared/lib/navigation/navigation-feedback.ts`
- `app/**/loading.tsx` для выбранных тяжелых маршрутов

## Критерии готовности
- progress bar не мигает на быстрых переходах
- работает и для ссылок, и для кнопок
- кнопка сразу показывает pending state и блокируется
- skeleton появляется на data-heavy routes
- нет blank state во время загрузки
- TypeScript strict
- без deprecated API
- корректная работа в light/dark theme

## Короткое резюме
После выполнения задачи в приложении появится единая система pending UX для навигации: top progress bar, SmartLink, helper-утилиты, корректный паттерн для кнопок и skeleton-загрузки для тяжелых страниц. В результате навигация станет понятнее, аккуратнее и будет восприниматься быстрее.
