# Бриф для агента: устранение race condition при переключении табов дня и загрузке тренировок / Kinescope player

## Контекст
В интерфейсе страницы дня пользователь переключается между табами дней. При клике на таб дня начинается загрузка тренировок и связанных данных, в том числе тяжелого контента, связанного с `Kinescope player`.

Если пользователь быстро переключается между табами:
- стартует новый запрос на следующий таб;
- предыдущий запрос может завершиться позже;
- в консоль и/или систему логирования летят ошибки от предыдущих запросов;
- при серии быстрых кликов возникает каскад ошибок;
- возможны лишние перерисовки, перемонтаж тяжелого player subtree и деградация UX.

Стек:
- React
- Next.js
- tRPC
- TanStack Query
- Kinescope player

## Проблема
Нужно проанализировать и предложить корректный паттерн обработки быстрых переключений между табами, чтобы:
- исключить ложные ошибки от отмененных или устаревших запросов;
- устранить race condition между старыми и новыми ответами;
- снизить лишнюю нагрузку от перемонтажа тяжелого player;
- сохранить быстрый и предсказуемый UX.

Агент должен:

## Целевое поведение
При быстром переключении табов должно происходить следующее:
- старый запрос либо корректно отменяется, либо его результат безопасно игнорируется;
- UI не показывает ошибку, если предыдущий запрос был прерван из-за смены таба;
- новый таб отображается стабильно и без дергания интерфейса;
- тяжелый `Kinescope player` не должен лишний раз монтироваться на каждый промежуточный клик;
- в мониторинг не должны улетать ложные ошибки по отмененным запросам.

## Важные вводные
- Ошибка отмененного запроса не должна трактоваться как продуктовая ошибка.
- Для UX важно не только отменить запрос, но и избежать лишнего перемонтажа player.
- Нельзя ограничиться только скрытием ошибок в консоли.
- Решение должно быть совместимо с декларативной моделью TanStack Query.

## Что нужно проверить в текущей реализации
Агенту нужно проанализировать:
1. Есть ли ручной `useEffect + fetch/setState`, который дублирует или обходит TanStack Query.
2. Как формируется `queryKey` для тренировок по дню.
3. Включен ли `abortOnUnmount` для tRPC query.
4. Используется ли `AbortSignal` на уровне transport/query function.
5. Как сейчас обрабатываются ошибки `AbortError` / `CanceledError`.
6. Монтируется ли `Kinescope player` сразу при смене таба.
7. Можно ли отделить:
   - активный день;
   - выбранную тренировку;
   - фактическое состояние воспроизведения player.
8. Есть ли лишние retry для отмененных запросов.
9. Есть ли ложные записи в Sentry / логах по ожидаемым cancel/abort.

## Базовые паттерны, которые нужно рассмотреть

### 1. Отмена предыдущего запроса
Рассмотреть использование `abortOnUnmount` в tRPC для query, связанных с табом дня.

Цель:
- при уходе с таба предыдущий запрос не должен продолжать жить как активный;
- отмененный запрос не должен считаться ошибкой бизнес-уровня.

### 2. Игнорирование abort как expected case
Нужно отделить:
- реальные network / server / parsing errors;
- expected abort при смене таба.

Цель:
- не показывать пользователю ложный error state;
- не шуметь в логах и мониторинге.

### 3. Один источник истины для активного дня
Данные тренировок должны зависеть от `activeDayId` через нормальный `queryKey`, а не через набор ручных императивных эффектов.

Цель:
- избежать stale overwrite;
- оставить поведение предсказуемым и декларативным.

### 4. Разделение состояния таба и состояния player
Рассмотреть декомпозицию:
- `activeDayId`
- `selectedWorkoutId`
- `playingWorkoutId`

Цель:
- не монтировать тяжелый player subtree при каждом переключении дня;
- загружать player только когда пользователь реально дошел до конкретной тренировки или начал просмотр.

### 5. Сохранение предыдущих данных на время загрузки
Рассмотреть `placeholderData` / `keepPreviousData`, чтобы не было резкого провала UI при переключении табов.

Цель:
- плавный UX;
- отсутствие лишнего мигания списка тренировок.

### 6. Prefetch соседних табов
Рассмотреть предзагрузку соседних дней или вероятных следующих вкладок.

Цель:
- ускорить perceived performance;
- уменьшить число видимых состояний загрузки.

### 7. Debounce / gating для тяжелого player
Рассмотреть отложенный монтаж player после краткой стабилизации выбора.

Цель:
- не создавать лишнюю нагрузку при серии быстрых переключений;
- уменьшить число ненужных инициализаций iframe/player.
