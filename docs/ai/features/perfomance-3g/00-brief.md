# Brief: Ускорение загрузки приложения на медленном интернете 3G

## Название задачи
Снизить время начальной загрузки приложения на медленном интернете 3G, в первую очередь для роутов platform
## Контекст
Сейчас приложение загружается слишком медленно в условиях слабой сети. На экране `day-page-client` observed time загрузки main-app составляет около **1.4 минуты**. По текущим наблюдениям загружаются **три layout-бандла**, ориентировочно:
- `546 KB`
- `531 KB`
- `1.4 KB`

Это означает, что пользователь слишком долго ждет первый полезный рендер, особенно на мобильных устройствах и плохом соединении. Для продукта это прямой удар по UX, удержанию и восприятию качества приложения.

## Цель
Найти и устранить основные причины медленной загрузки main-app на `day-page-client` в условиях slow 3G и добиться заметного сокращения времени до интерактивности и первого полезного отображения.

## Основная проблема
На слабой сети страница `day-page-client` загружает слишком большой объем JS на старте. Вероятные причины:
- лишний JS попадает в initial load
- клиентские layout или providers тянут тяжелые зависимости слишком рано
- есть избыточная вложенность layout
- код плохо разделен по route-level boundaries
- тяжелые client components и shared зависимости попадают в critical path
- отсутствует или недостаточно агрессивен lazy loading / dynamic import
- возможен лишний hydration cost

## Что нужно сделать

### 1. Исследовать источник большого initial JS
Нужно определить:
- какие именно чанки входят в initial load для `day-page-client`
- какой layout или provider тянет больше всего JS
- какие client components попадают в critical rendering path
- какие зависимости реально нужны для first render, а какие можно отложить

### 2. Проверить composition layout и providers
Нужно проанализировать:
- сколько layout реально участвуют в рендере `day-page-client`
- какие из них являются client components
- какие providers смонтированы слишком высоко
- можно ли опустить часть client-only логики ниже по дереву
- можно ли локализовать тяжелые providers только там, где они действительно нужны

### 3. Сократить initial bundle для `day-page-client`
Нужно вынести из начальной загрузки все, что не требуется для первого экрана:
- второстепенные виджеты
- модальные окна
- тяжелые интерактивные блоки
- аналитические и вспомогательные клиентские модули
- редко используемые UI-ветки
- большие библиотеки, если они не нужны для first paint

### 4. Проверить route-level и component-level code splitting
Нужно убедиться:
- что тяжелые блоки подгружаются только по необходимости
- что dynamic import используется там, где это оправдано
- что нет ситуации, когда shared-компонент случайно затягивает тяжелую библиотеку во все маршруты
- что `day-page-client` не наследует лишний код от верхних layout

### 5. Проверить hydration cost
Нужно определить:
- нет ли на `day-page-client` слишком большого числа client components
- можно ли часть UI сделать server-first
- можно ли уменьшить объем клиентской логики на первом экране
- нет ли дорогих эффектов, подписок и инициализаций сразу при mount

### 6. Подготовить план оптимизации и внедрить phase 1
После исследования нужно:
- перечислить главные bottleneck
- оценить вклад каждого bottleneck в размер initial JS или время загрузки
- предложить конкретные изменения
- внедрить наиболее выгодные улучшения первой очереди

## Что считать результатом
Задача не только в общем рефакторинге, а в реальном улучшении загрузки на slow 3G.

После выполнения нужно получить:
- меньше initial JS для `day-page-client`
- меньше JS в layout critical path
- более быстрый first render
- более быстрое появление usable UI
- сокращение времени загрузки main-app по сравнению с текущим baseline около 1.4 минуты

## Что важно проверить в первую очередь
Приоритет анализа:
1. `app/layout` и вложенные layout, участвующие в `day-page-client`
2. глобальные providers
3. тяжелые client-only зависимости
4. route-specific components экрана `day-page-client`
5. shared UI/feature modules, которые попадают в initial bundle без необходимости

## Non-goals
- не делать бессистемный массовый рефакторинг всего приложения без связи с bottleneck
- не оптимизировать сначала второстепенные маршруты, если проблема подтверждена на `day-page-client`
- не ухудшать UX ради формального уменьшения bundle size
- не выносить логику в dynamic import без проверки, что это реально улучшает critical path
- не ломать SSR, App Router composition и текущую бизнес-логику

## Ожидаемые артефакты от агента

### A. Research
Короткий отчет:
- какие layout участвуют в загрузке `day-page-client`
- какие чанки формируют initial load
- какие client components и зависимости самые тяжелые
- где именно main-app теряет время на slow 3G

### B. Findings
Список основных причин с приоритетом:
- root cause
- влияние
- примерный expected impact after fix

### C. Optimization plan
Пошаговый план:
- что исправить сначала
- что можно вынести из critical path
- что перевести в lazy/dynamic loading
- какие providers или components нужно опустить ниже
- что можно перевести в server-first rendering

### D. Implementation phase 1
Сделать наиболее выгодные улучшения первой очереди без поломки архитектуры.

### E. Verification
После изменений показать:
- что изменилось по initial JS
- какие layout/chunks стали легче
- сократилось ли время загрузки на slow 3G
- нет ли регрессий по рендерингу и UX

## Acceptance criteria

### Performance
- время загрузки `day-page-client` на slow 3G заметно снижено относительно baseline
- initial JS для `day-page-client` уменьшен
- main-app не тянет лишние layout-бандлы без необходимости
- первый полезный контент появляется раньше

### Architecture
- оптимизации не завязаны на хаки
- не добавлены случайные дубли или обходные решения
- соблюден текущий app-router подход
- не ухудшена поддерживаемость кода

### UX
- пользователь быстрее видит usable интерфейс
- нет лишних blank states
- нет ухудшения навигации и интерактивности после загрузки

## Дополнительные указания агенту
При анализе не ограничиваться только bundle size. Нужно смотреть шире:
- размер JS
- число client components
- hydration cost
- composition layout
- providers
- тяжелые shared imports
- dynamic import opportunities
- server/client boundaries

Важно найти именно root cause, а не просто уменьшить цифры локально.

## Короткое резюме
Нужно исследовать, почему `day-page-client` загружается около 1.4 минуты на slow 3G, определить вклад трех layout-бандлов и всего initial JS, затем убрать лишний код из critical path и внедрить первую очередь оптимизаций, чтобы приложение ощутимо быстрее загружалось на слабом интернете.
